ArcLoom Specification v1.2 — Binary‑Free SPU (Option A: Ternary Differential Signaling)
=======================================================================================
Date: 2026-01-07
Status: Full replacement; Option A physical layer + performance/feel metrics.

Core premise (normative)
========================

Core premise (normative, non‑negotiable):
ArcLoom SHALL be binary‑free at the architectural primitive level:
• No binary ISA, no instruction fetch/decode/execute, no program counter model.
• No requirement that internal state be representable as bits.
• Discrete control/state, when used, SHALL be ternary or multi‑level (≥3 stable states).
• Binary interoperability SHALL be handled outside the SPU boundary via an external story‑ingestion gateway.

Normative references
====================
UF Spec v1.4.0 is normative. IRF and UFCP/Not‑Math are informative framing/constraints.

0. Scope
========
Defines ArcLoom as deterministic field→gate→DSF→commit→motif memory→output, binary‑free primitives, Option A physical layer.

1.1–1.3 Conformance clauses
===========================

1.1 Determinism classes (normative)
• Class A (Reference‑Deterministic): identical inputs/config ⇒ byte‑identical traces in reference model.
• Class B (Device‑Deterministic): identical inputs/config ⇒ identical gate boundaries and DSF within specified tolerances under bounded device noise.

1.2 Binary‑Free compliance (normative)
BF‑1: SPU SHALL contain no binary instruction execution engine (no ISA, no PC, no fetch/decode/execute).
BF‑2: SPU SHALL not require state to be representable as bits.
BF‑3: Any discrete internal control/state SHALL be ternary or multi‑level (≥3 stable states).
BF‑4: Safety/governance SHALL NOT depend on a binary firmware loop.
BF‑5: Any binary computing device used for interop SHALL be outside the SPU boundary (external gateway).

1.3 Option A compliance (normative)
OA‑1: On‑chip interconnect and I/O signaling SHALL use ternary differential signaling (Clause 4).
OA‑2: Ternary receivers SHALL classify into exactly three states with declared thresholds.
OA‑3: Line coding SHALL be balanced to reduce DC drift (recommended: balanced ternary).

2.1–2.2 Block model
===================

2.1 Blocks (normative)
A) Coupler Array (CA) — sensor/actuator interfaces and stream conditioning
B) Ternary Differential Signaling Fabric (TDSF) — Option A physical layer
C) Kernel Fabric (KF) — UF L0–L4 operator pipeline emitting DSF per gate
D) Loom Fabric (LF) — settle/commit, motif formation, cue‑driven recall
E) Krimelacks Memory (KM) — folded structural memory (multi‑level NVM + caches)
F) Governance and Safety (GS) — hardening triggers/actions, SafeMode, output holds
G) Domain Adapters (DA) — L5 adapters mapping DSF↔domain outputs

2.2 Interfaces (normative)
• Input: F(t) ∈ R^d, optional relevance r(t)
• Kernel output: DSF_k per gate
• Loom artifacts: motifs/strands/landmarks
• Output: bounded y(t) with SafeHold/SafeMode rules

3.1–3.3 Maths used (summary)
============================

3.1 Normative kernel equations (summary; see UF Spec v1.4.0 for full definition)

L0:
∆F(t)=F(t)−F(t−∆t)
σ(t)=(1/W) Σ_{i=0}^{W−1} ||F(t−i∆t) − \bar F_W(t)||^2
κ(t)=||F(t+∆t)−2F(t)+F(t−∆t)||
Negative space:
N(t)=1 if σ(t)<σ_min ∧ ||∆F(t)||<δ_min ∧ κ(t)<κ_min else 0
SEV(t)=(F(t),∆F(t),σ(t),κ(t),r(t),N(t))

L1:
D(t)=α1||∆F(t)||+α2σ(t)+α3κ(t)
D(t)≥τ_D ⇒ boundary; gate opens when D(t)<τ_D

L4 DSF (direction is ternary):
D_k ∈ {−1,0,+1} from ∆R(k) thresholds
DSF_k=(D_k,M_k,R_k,U*_k,C_k,P_k,B_k)

3.2 Mandatory invariants (examples; normative)
0 ≤ R(k) ≤ 1
0 ≤ U_k ≤ 1
C_k ≤ C_max
B_min ≤ B_k ≤ B_max

3.3 SafeMode (normative)
SafeMode = 1 ⇔ S(UF) < S_min
In SafeMode, DSF is neutralized and outward actions are held.

4.1–4.5 Option A rules
======================

4.1 Ternary symbol set (normative)
T = { −1, 0, +1 }  (“balanced trits”)

4.2 Differential encoding (normative)
Each trit SHALL be represented by a differential level Vdiff on a pair (P,N):
t=−1 ⇒ Vdiff ≤ V_L
t= 0 ⇒ V_M_low < Vdiff < V_M_high
t=+1 ⇒ Vdiff ≥ V_H

The implementation SHALL declare (V_L, V_M_low, V_M_high, V_H) and guarantee
noise margins under stated operating conditions.

4.3 Receiver classification (normative)
Receivers SHALL implement triple‑threshold classification producing exactly one of {−1,0,+1}.
Classification SHALL be deterministic for any Vdiff and shall define tie‑break rules on thresholds.

4.4 Balanced coding (recommended, normative if declared)
To mitigate DC drift and baseline wander, streams SHOULD be balanced:
over a window Wc, Σ t_i ≈ 0.

If balanced coding is used, the encoder/decoder SHALL be deterministic and SHALL NOT use binary framing.

4.5 Timing and framing (normative)
The fabric SHALL declare a timing model:
• event‑driven (symbol emitted on change) or
• clocked (symbols at ∆t intervals).
Timestamp monotonicity is mandatory per UF L0 input health rules.

5.1–5.3 I/O + time rules
========================

5.1 Input health verification (normative)
Inputs SHALL be rejected if:
• timestamp monotonicity fails,
• ||∆F(t)|| > ∆max,
• σ(t) ∉ [0, σmax),
• F(t) ∉ D ⊂ R^d.
(These are the UF L0 input health verification rules.)

5.2 Negative‑space checks (normative)
The negative‑space operator N(t) SHALL be computed and preserved downstream.
Absence is treated as first‑class structural information.

5.3 Time integrity (normative)
ArcLoom SHALL maintain a monotone time index within the SPU boundary.
If an external timing reference is used, it SHALL be sanity-checked and bounded.

6.1–6.3 Kernel rules
====================

6.1 Operator implementation (normative)
Each UF operator SHALL be implemented as a deterministic function with bounded outputs.

6.2 Bounded DSF variation (normative)
Implementations SHALL satisfy DSF bounded variation:
||DSF_k − DSF_{k−1}|| ≤ ε_DSF unless |∆R(k)| > ε_D.

6.3 SafeMode enforcement (normative)
SafeMode SHALL activate iff S(UF) < S_min.
In SafeMode, DSF SHALL be neutralized and outputs held.

7.1–7.4 Loom rules
==================

7.1 Commit rule (normative)
Commit SHALL occur only if:
• SafeMode = 0
• Dominance ≥ p_commit
• Determinism ≥ det_commit
• Commit rate ≤ max_commits_per_time

7.2 Determinism metric (normative)
The implementation SHALL declare Determinism(k) ∈ [0,1] as a function of DSF/history.
(Informative: IRF-style entropy-drop / determinism index may be used as a metric.)

7.3 Recall rule (normative)
Given a cue q, recall SHALL return:
m* = argmax_m Similarity(q,m)
Similarity MUST be deterministic and bounded.

7.4 Fold locality (normative)
Memory adjacency SHALL be updated such that co‑activation increases proximity.

8.1–8.3 Memory rules
====================

8.1 Multi‑level storage requirement (normative)
Motif storage cells SHALL support ≥3 stable states/cell, or motifs SHALL be encoded in a way that preserves ternary/multi‑level semantics without binary dependence.

8.2 Bounded growth (normative)
Implementation SHALL enforce motif_budget and strand_budget; decay MUST be deterministic.

8.3 Export controls (normative)
Export and externalization SHALL be policy‑gated and logged.

9.1–9.3 Output rules
====================

9.1 Output constraints (normative)
Each output SHALL declare:
• max magnitude
• max rate
• SafeHold behavior
• SafeMode behavior (usually disable)

9.2 SafeHold (normative)
When hardening triggers or SafeMode is active, outputs SHALL be held/disabled deterministically.

9.3 No binary framing (normative)
If discrete output symbols are used, they SHALL be ternary/multi‑level under Option A.

10.1–10.3 Governance rules
==========================

10.1 Hardening trigger conditions (examples; normative if UF spec referenced)
HTCs include: Ck>Cmax, Uk>Umax, breathing out of range, schema growth violations,
stability collapse S(UF)<Smin, DSF instability, drift violations.

10.2 Hardening actions (examples)
Gate suppression gk=0
DSF neutralization to a hold vector
Breathing contraction
Schema freeze/merge

10.3 SafeMode (normative)
SafeMode = 1 ⇔ S(UF) < Smin
In SafeMode: decision dynamics freeze; DSF neutralized; schema formation suspended; outputs held.

11.1–11.6 Performance metrics
=============================

11.1 Reality‑to‑Action Latency (R2A) (normative)
R2A = t_out_change − t_in_change
Report median, p95, p99.

11.2 Latency Jitter (normative)
Jitter = Var(R2A) over a test run; also report p99−p50.

11.3 Semantic Bandwidth Ratio (SBR) (normative)
Let BW_in be raw sensor stream bandwidth (samples/sec × bytes/sample).
Let BW_out be DSF+output stream bandwidth.
SBR = BW_out / BW_in.
Lower SBR indicates less “data tax” and more structural compression.

11.4 Commit Density (CD) (normative)
CD = commits per minute (or per 1e6 samples). Must remain ≤ declared max_commits_per_time.

11.5 Stability Events (SE) (normative)
SE = count of:
• hardening actions invoked
• SafeMode entries
• recovery time from SafeMode

11.6 Energy per Meaningful Decision (EMD) (recommended)
If energy is measured:
EMD = Joules / (number of DSF updates + commits).

Annex A — Build likelihood and worth it (informative)
=====================================================

A.1 What is straightforward vs hard

Straightforward (high likelihood):
• Implement UF L0–L4 as deterministic operator blocks (streaming math; bounded outputs).
• Enforce invariants, hardening actions, and SafeMode as explicit state logic.
• Use DSF as a low-bandwidth interface into domain adapters.

Hard (medium likelihood, engineering-heavy):
• Ternary differential fabric at scale (noise margins, thresholds, testing).
• Multi-level NVM integration tuned for stable ≥3-state storage and endurance.
• Calibration that remains binary-free in its control semantics.

Very hard (low likelihood initially):
• Building a general-purpose “whole computer ecosystem” that outperforms CPUs on classic tasks like exact arithmetic, random memory addressing, and arbitrary symbolic transforms.

A.2 When ArcLoom is “better” (not a toy)

ArcLoom is worth it if, on real tasks, it achieves measurable wins on:
• Lower R2A latency and lower jitter (robots “feel” smoother and safer).
• Lower SBR (less data marshaling / less bus traffic for the same outcome).
• Fewer stability failures under noise/shock (bounded behavior, SafeMode instead of crash).

A.3 When ArcLoom is not better

If your workload is primarily:
• exact arithmetic over large integers,
• random-access data structures,
• classic OS/app workloads,
then CPUs will likely remain superior.

A.4 Recommended first success criterion (explicit)

RECOMMENDED: prove “worth it” on robotics-style tasks first:
A 10× reduction in p99 control latency jitter OR a 10× reduction in BW moved per stable control outcome (SBR) versus a CPU-based stack.

Annex B — Benchmark protocol template (informative)
===================================================

B.1 Step-response benchmark
• Apply a known step change on one sensor channel.
• Measure R2A latency distribution and jitter.

B.2 Flood benchmark
• Apply high-bandwidth continuous variation (e.g., vibrating IMU input or moving visual pattern).
• Measure SBR and thermal/energy (optional).

B.3 Shock benchmark
• Inject adversarial spikes (variance/curvature) consistent with UF shock tests.
• Verify SafeMode triggers iff S(UF) < S_min and recovery returns to baseline.
